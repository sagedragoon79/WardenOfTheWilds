## The Bug

Trap Master Hunter Lodges were accumulating meat far past the 200 cap (user's save: **1811 meat**). The stacked meat was invisible to nearby smokehouses, wagons, and cellars — no haul tasks were generated against it. BGH path was unaffected.

**User's key insight that cracked the case:** *"Could it have anything to do with meat auto delivering to the lodge? Like the bear bonus?"* — yes.

## Root Cause (confirmed via decompile)

A `HunterBuilding` has **two storage pools**:
- `base.storage` — the OUTPUT pool. `ProduceItems` writes here. `CheckWorkAvailabilityForItem` queries this (and ONLY this) to register the `HasItemMeat` work bucket that drives logistics.
- `manufacturingStorage` — the carcass INPUT pool.

`GetItemCountFromAllStorages` sums **both**, which is what the building UI displays — so to the player, the meat looked normal-ish, just stuck.

Both bear-bonus patches (`BearBonusYieldPatches.cs` and `TrapMasterBearPatches.cs`) had been depositing into `cabin.manufacturingStorage`. Items were therefore:
1. Counted in the visible total (UI showed 1811)
2. Invisible to `HasItemMeat` (logistics never woke up)
3. Free to bypass the per-item cap (the cap is enforced on the OUTPUT side)

Compounding factor: `ReservableItemStorage.AddItems` returns the **new total**, not a delta — so deposit logs printed misleading "+1500 meat" numbers when 1280 already existed.

## Fixes Shipped

**v1.0.7 — Diagnostic build.** Added `TrapperMeatDiagnosticLoop` in `HunterCabinEnhancement.cs` (gated by `DiagnosticsEnabled`) so we could see exactly where overflow was landing without guessing.

**v1.0.8 — Targeted fix.**
- Both bear-bonus patches now write to `cabin.storage` (output pool).
- Added `DrainStuckBearBonusItems()` migration in `HunterCabinEnhancement.cs` that runs once per cabin per session at attach, transferring stuck Meat/Hide/Tallow from `manufacturingStorage` → `base.storage` for existing saves.

**v1.0.9 — Hardening.** Added `DepositClamped` helper in `TrapMasterBearPatches.cs`:
- Clamps deposits to `HunterCabinOutputStorageCap` (so bonuses can't push the cabin past its per-item cap if the building still has total room).
- Computes the actual delta from before/after counts, so logs report accurate "+N" numbers.
- Logs forfeited overflow when the cabin is at cap.

`BearBonusYieldPatches.cs` was updated to use the same `DepositClamped` helper.
